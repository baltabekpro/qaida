# Flutter frontend guide for Qaida

## Зачем нужен этот документ

Этот документ объясняет, как Flutter-разработчику работать с нашими дизайн-референсами из папки `design`, чтобы переносить их в приложение без потери визуальной логики, паттернов поведения и консистентности.

## Источник истины

Основной источник истины по интерфейсу находится здесь:

- `design/index.html` — дизайн-борд, UI kit, токены, паттерны, reference screens
- `design/qaida_1/code.html` - `design/qaida_17/code.html` — отдельные экранные референсы

Важно:

- `design/index.html` нужен для понимания общей дизайн-системы
- каждый `qaida_*` экран нужен для конкретной реализации сценария
- HTML-референсы не являются production-ready кодом для Flutter, это визуальная спецификация
- если есть конфликт между абстрактным компонентом и реальным экраном, приоритет у реального пользовательского сценария, но без нарушения общей системы

## Как читать структуру дизайнов

### 1. Сначала смотрим на board

Перед началом работы открой `design/index.html` и посмотри:

- Foundation
- Typography
- Core Components
- Cards & Modules
- Popups & Overlays
- Notifications
- Filter System
- Reference Screens

Это даёт понимание:

- какие цвета и поверхности считать базовыми
- какой уровень контраста считается нормальным
- какие радиусы, тени и формы считаются системными
- какие состояния кнопок, табов, карточек и сохранения допустимы

### 2. Потом смотрим конкретный сценарий

После board переходи к нужному `qaida_*` экрану.

Примерный маппинг:

- `qaida_2` — онбординг
- `qaida_3` — главная
- `qaida_4` — фильтр, шаг 1
- `qaida_5` — персональные рекомендации
- `qaida_18` — поиск и discovery state
- `qaida_7` — сохранённые места
- `qaida_19` — список коллекций
- `qaida_20` — детали коллекции
- `qaida_9` — уведомления
- `qaida_21` — настройки уведомлений
- `qaida_10` — bottom sheet
- `qaida_11` — empty state
- `qaida_12` — confirm dialog
- `qaida_13` — permission popup
- `qaida_14` — сохранение без текстового toast
- `qaida_22` — редактирование данных профиля
- `qaida_15` - `qaida_17` — остальные шаги фильтра

Если реализуется целый flow, не надо переносить экран по одному в отрыве от остальных. Нужно сначала понять весь сценарий целиком.

## Главные принципы интерфейса

### 1. Чистый интерфейс

Мы избегаем лишнего текста, если состояние можно показать:

- цветом
- иконкой
- fill-состоянием
- мягкой микроанимацией
- сменой поверхности

Пример:

- сохранение места подтверждается состоянием флажка, а не длинным snackbar с текстом

### 2. Низкий визуальный шум

Нельзя делать интерфейс слишком контрастным.

Что это значит на практике:

- не использовать слишком тяжёлые тени
- не ставить яркие толстые outline без необходимости
- не делать выбранные карточки слишком синими
- не перегружать экран вторичными подписями
- не делать сразу несколько акцентных зон одинаково яркими

### 3. Консистентность важнее локальной красоты

Если компонент на одном экране выглядит красиво, но выбивается из системы, его надо приводить к общему паттерну.

Особенно это касается:

- табов и segmented controls
- CTA-кнопок
- карточек фильтров
- empty states
- popups и overlays
- флажков сохранения

### 4. Спокойный премиальный тон

Интерфейс должен ощущаться:

- лёгким
- аккуратным
- современным
- не кричащим
- ближе к curated lifestyle product, а не к utility app с агрессивными CTA

## Дизайн-токены, которые нужно вынести в Flutter

Ниже не попытка пиксель-в-пиксель скопировать HTML, а базовые системные значения, которые должны жить в Flutter theme/tokens.

### Цвета

Рекомендуемый стартовый набор токенов:

- `primary = #2463EB`
- `primarySoft = #7DB7FF`
- `textPrimary = #162033`
- `textSecondary = #6A778B`
- `surface = #FFFFFF`
- `surfaceSoft = #F8FBFF`
- `surfaceMuted = #F2F6FC`
- `borderSoft = rgba(15, 23, 42, 0.08)`
- `borderStrong = rgba(15, 23, 42, 0.14)`
- `successSoft`, `warningSoft`, `infoSoft` — выводить из контекста экрана, не делать кислотными

Для dark theme ориентиры такие:

- `darkBg = #09111F`
- `darkSurface = #0F172A`
- `darkSurfaceSoft = #13203A`
- `darkText = #E8EEF8`
- `darkTextSecondary = #94A3B8`

### Радиусы

Системные радиусы:

- `radiusXL = 32`
- `radiusLG = 24`
- `radiusMD = 18`
- `radiusSM = 14`
- `radiusFull = 999`

Важно:

- не смешивать слишком много разных радиусов на одном экране
- у карточек и bottom sheet использовать большие мягкие радиусы
- у внутренних controls радиусы меньше, но всё ещё округлые

### Тени

Тень должна быть мягкой и глубокой, но не грязной.

Базовые ориентиры:

- soft shadow: низкая непрозрачность, большой blur
- primary shadow: только для основного CTA или активного состояния
- overlay shadow: глубже, чем у карточек, но не чёрная лужа

Не нужно буквально копировать CSS-тень 1 в 1. Нужен тот же визуальный характер.

## Типографика

### Шрифты

В референсах используется:

- Inter — основной UI-шрифт
- Manrope — display/board usage

Для мобильного Flutter-интерфейса можно оставить:

- Inter как основной шрифт приложения
- Manrope использовать только если это подтверждено продуктово и не ломает общий тон

### Иерархия текста

Принцип:

- заголовки должны быть плотными, но не тяжёлыми
- вторичный текст должен быть мягким и читабельным
- UI labels не должны спорить с контентом

Практическое правило:

- не злоупотреблять `FontWeight.w800`
- для большинства крупных заголовков подойдут `w600` или `w700`
- вторичный текст чаще всего `w400` или `w500`
- CTA обычно `w600`, а не обязательно `w800`

Для фильтров мы уже пришли к более спокойной типографике. Это нужно сохранить и в Flutter.

## Иконки

Используются Material Symbols.

Правила:

- иконки важнее декоративных emoji
- если состояние активное, можно использовать `fill`
- подтверждение действия лучше делать через смену состояния иконки, чем через отдельный текст
- размер иконок должен соответствовать плотности блока, а не жить сам по себе

Во Flutter это значит:

- либо Material Symbols package
- либо близкий по пластике набор иконок, но последовательно по всему приложению

## Как переносить HTML-референсы во Flutter

### Не переносить верстку буквально

Нельзя просто механически воспроизвести HTML блок за блоком.

Нужно выделять:

- screen scaffold
- app bar
- section patterns
- reusable cards
- chips
- buttons
- bottom navigation
- overlay patterns
- empty states
- stateful icons

### Что нужно сделать сначала

Перед кодингом нового экрана:

1. Определи сценарий и его состояние.
2. Найди все связанные референсы в `design/qaida_*`.
3. Выпиши reusable widgets.
4. Выпиши design tokens, которые экран использует.
5. Определи интерактивные состояния: default, pressed, selected, disabled, saved, empty, loading.

### Что должно стать Flutter-компонентами

Минимальный набор reusable widgets:

- `QaidaScaffold`
- `QaidaAppBar`
- `PrimaryButton`
- `SecondaryButton`
- `FilterOptionCard`
- `SegmentedPillTabs`
- `SavedBookmarkButton`
- `PlaceCard`
- `NotificationCard`
- `BottomSheetActionPanel`
- `EmptyStateBlock`
- `SelectionChip`

Если одинаковый паттерн появился на 2+ экранах, он должен стать переиспользуемым widget.

## Рекомендованная структура Flutter UI-слоя

Пример:

```text
lib/
  app/
    theme/
      app_colors.dart
      app_radius.dart
      app_shadows.dart
      app_typography.dart
      app_theme.dart
  features/
    home/
    filters/
    saved/
    notifications/
    recommendations/
  shared/
    widgets/
      buttons/
      cards/
      chips/
      navigation/
      overlays/
      states/
```

Важно:

- токены не должны лежать внутри feature-модулей
- общие UI-компоненты не должны зависеть от конкретного backend response shape
- экран не должен вручную собирать стили в каждом widget tree

## Правила по конкретным паттернам

### Сохранение

Актуальный паттерн:

- отдельный текстовый toast не обязателен
- основной сигнал — сам флажок
- допустимы:
  - смена цвета
  - fill state
  - короткая пульсация
  - лёгкое кольцо вокруг иконки
- недопустимы:
  - тяжёлый snackbar с большим количеством текста
  - навязчивая анимация
  - сразу несколько подтверждений одного действия

### Фильтры

Фильтр — это не техническая форма, а спокойный guided flow.

Правила:

- не делать selected state слишком контрастным
- шаги должны отличаться логикой, а не ломать систему
- карточки должны быть читаемыми, но не кричащими
- progress bar должен быть вторичным элементом
- CTA — заметный, но не самый тяжёлый объект на экране

### Saved / Notifications tabs

Для разделения экранов используем pill-tabs, а не случайный микс разных табов.

Правила:

- активный pill — залит primary
- неактивный pill — светлая поверхность с тонкой границей
- размеры, padding и radii должны совпадать между экранами

### Popups and overlays

Используем только те overlays, которые действительно помогают сценарию.

Если состояние можно показать внутри элемента, не надо создавать отдельный overlay.

## Процесс работы Flutter-разработчика

### На старте задачи

1. Открыть board и нужные reference screens.
2. Проверить, существует ли уже такой паттерн в приложении.
3. Если существует, переиспользовать его, а не создавать новый.
4. Если не существует, сначала выделить новый reusable component.

### Во время реализации

1. Сначала собрать theme и tokens.
2. Потом собрать reusable widgets.
3. Только потом собирать screen.
4. Проверять экран на light и dark theme.
5. Проверять экран на длинных текстах и узких девайсах.

### Перед PR

Чеклист:

- экран визуально совпадает с референсом по ритму, а не только по цвету
- веса шрифта не стали тяжелее без причины
- отступы и радиусы системные
- selected/active states не слишком контрастные
- нет лишнего текста там, где достаточно icon/state feedback
- dark theme не сломана
- компонент переиспользуем, если паттерн повторяется

## Когда нужно уточнение у дизайна

Нужно спрашивать, если:

- неясно, это отдельный новый паттерн или вариация существующего
- состояние элемента не показано в референсе
- непонятно, нужен ли overlay или достаточно inline feedback
- контент в реальном приложении длиннее, чем в макете
- один и тот же паттерн выглядит по-разному в двух местах

## Чего делать не надо

- не копировать HTML буквально в Flutter мышлением `Container на Container`
- не придумывать свои новые radii, тени и табы без причины
- не усиливать контраст только ради ощущения "более современно"
- не заменять системные иконки на emoji
- не добавлять текстовые подтверждения там, где уже хватает состояния иконки
- не делать локальные одноразовые widgets для повторяющегося паттерна

## Быстрый summary для разработчика

Если коротко:

- `design/index.html` — это система
- `design/qaida_*` — это сценарии
- Flutter должен переносить не HTML, а паттерны
- главный критерий качества — консистентность, чистота и спокойный контраст
- если можно показать состояние без текста, лучше показать без текста
